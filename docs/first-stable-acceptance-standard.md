# First Stable Acceptance Standard

## Scope

This document defines the first stable acceptance flow for the Easy stack:

- `EasyProxy`
- `EasyEmail`
- `EasyBrowser`
- `EasyProtocol`
- `EasyRegister`

The acceptance target is not "full business success". The target is a repeatable
blank-host-style deployment and a verified end-to-end runtime path that reaches
the real business blocking point.

## Required Preconditions

1. All acceptance-related code must be committed and pushed to the public GitHub
   repositories before the run starts.
2. GitHub-hosted publish workflows must be re-triggered for the current code
   revision so fresh GHCR images and release artifacts exist for this acceptance
   round.
3. Local runtime data must be preserved.
   In particular, do not delete data under:
   - `C:\Users\vmjcv\.cli-proxy-api\free`
   - `C:\Users\vmjcv\.cli-proxy-api\team`

## Required Deployment Style

1. Use a brand-new empty host working directory for the acceptance run.
2. Do not clone the full repositories into that host working directory.
3. Only download each repository root `deploy-host.ps1` script into the blank
   working directory.
4. Use those downloaded root deploy scripts to bootstrap and deploy the services.
5. When a repository uses encrypted import-code distribution, obtain the current
   import-code artifact generated by the hosted publish workflow, decrypt it
   locally, and use it as the deployment input.
6. The owner private key used to decrypt those artifacts may be either a raw
   Curve25519 private key or a stable passphrase string, but every repository
   in this stack must derive the same owner public key from that private input
   before publication.
7. Prefer GHCR images produced by the fresh hosted workflow run. Do not reuse
   locally built images as the acceptance source of truth.

## Repository-Specific Blank-Host Entry Rules

1. `EasyProxy` GHCR acceptance deployment must use the GHCR project path:
   - `deploy-host.ps1 -Project easyproxy-ghcr`
   - Do not use `-Project easyproxy` for this acceptance, because that path
     can still fall back to local compose/build behavior.
2. `EasyProtocol` acceptance deployment must keep:
   - compose project: `easy-protocol`
   - gateway container: `easy-protocol`
   - sibling provider containers under the same compose project
3. `EasyRegister` acceptance deployment must keep:
   - compose project: `easy-register`
   - mixed single-runtime deployment
   - explicit concurrency env:
     - `REGISTER_WORKER_COUNT=10`
     - `REGISTER_MAIN_CONCURRENCY_LIMIT=5`
     - `REGISTER_CONTINUE_CONCURRENCY_LIMIT=2`
     - `REGISTER_TEAM_CONCURRENCY_LIMIT=1`

## Required Local Cleanup Before Redeploy

1. Remove local Easy-stack runtime containers that would interfere with the new
   acceptance deployment.
2. Remove local Easy-stack images that would allow the deployment to silently
   reuse stale local builds.
3. Do not delete bind-mounted host data directories.

## Required Runtime Topology Checks

After deployment, confirm:

1. `EasyProxy`, `EasyEmail`, `EasyBrowser`, `EasyProtocol`, and `EasyRegister`
   are all running from the fresh acceptance deployment.
2. `EasyProtocol` uses the canonical compose project `easy-protocol`.
3. `easy-protocol` is the gateway container.
4. Provider children such as `easy-protocol-python-001` are created under the
   same `easy-protocol` compose project.
5. `EasyBrowser` and `EasyProtocol` both support manager-layer dynamic scaling
   of sibling executor/provider containers.

## Required EasyRegister Runtime Checks

1. `EasyRegister` runs as a single mixed runtime.
2. Total worker concurrency remains `10`.
3. Default per-flow concurrency caps remain:
   - `main = 5`
   - `continue = 2`
   - `team = 1`
4. `continue` must still run when no valid `team` mother credential exists.
5. In the no-mother path, `continue` must skip invite/revoke steps instead of
   stalling the flow.

## Acceptance Success Condition

The acceptance run is considered successful when all of the following are true:

1. The five services are deployed from the blank host working directory using
   the root deploy scripts.
2. `EasyRegister` can start and dispatch real `main` tasks against the deployed
   `EasyProtocol` and `EasyBrowser` stack.
3. The `main` flow can advance through the real registration path instead of
   failing on local deployment, mount, service-discovery, or internal routing
   issues.
4. If the flow ultimately reaches a real business blocker such as:
   - `add_phone`
   - `authorize_continue_rate_limited`
   - `unsupported_country`

   then the acceptance still counts as passed, because the deployment/runtime
   chain is verified and the remaining blocker is a genuine upstream business
   result instead of a local architecture defect.

## Failure Classification

The acceptance run does **not** pass if the observed failure is caused by:

1. Missing or stale GHCR image publication.
2. Root deploy script failure.
3. Missing import-code/bootstrap material when the repo requires it.
4. Wrong internal service aliasing or wrong API key injection.
5. Wrong shared bind mounts such as mismatched `/shared/register-output`.
6. `EasyProtocol` or `EasyBrowser` failing to create dynamic child containers.
7. `EasyRegister` failing before the real business flow due to local orchestration
   or container-runtime defects.
